Agent Task Timeline

Decision anchor for iOS and Android. Not a feature spec — a shared source of truth so we stop re-asking the same questions.

Principles

1 Show the outcome, not the process

Users want to know: what did I ask, what happened, did it work?

2 One piece of info, one place

Never show the same content twice. Extract and deduplicate.

3 Bursts collapse, sessions don't

Tool call groups are collapsible. Session boundaries are lightweight dividers.

4 Platform parity over flair

Visual styling can differ. Information architecture must not.

5 Terminal = frozen, active = live

Summaries only appear for finished sessions. Never for something still running.

Core Concepts

📋
Task

A user request to a Copilot agent. Contains one or more sessions. "Fix the auth bug" is a task.

→
⚡
Session

A single agent execution. Each session has its own event stream, state, and lifecycle.

→
⚛
Event

An atomic unit — messages, tool calls, permissions. The timeline is built from these.

Timeline Layout

The timeline is a flat vertical stream. No per-session cards. Sessions flow into each other, separated by lightweight dividers.

Key rules

  • User bubble — always visible at the top of each session
  • Summary — always visible at the bottom (terminal sessions only)
  • Working indicator — bottom of timeline when session is in_progress
  • Tool call bursts — the only collapsible unit

Work Bursts

Consecutive tool calls group into collapsible "bursts" to reduce noise.

Grouping rules

  1. Minimum 3 consecutive tools to form a group
  2. "task" tool breaks the burst (agent launches are significant)
  3. User messages break the burst (new conversational turn)
  4. Permissions break the burst (interaction points)
  5. Last burst auto-expands if session is in-progress or no trailing message follows
⚠️ Platform gap

iOS implements full burst grouping. Android only groups consecutive same-name tools — doesn't implement the broader work burst pattern yet.

Multi-Session

When a task has multiple sessions (follow-ups, different surfaces), they're separated by inline dividers — not cards.

Divider format

Continued from <origin> · <model> · <time>

Origin maps from producer: copilot-cli → "CLI", copilot-web → "web", copilot-vscode → "VS Code"

Rules

  • Flat timeline — sessions are not collapsible units
  • Chronological order, oldest first
  • Each session is self-contained (no cross-session leaking)
  • Only the first session has no divider
⚠️ Platform gap

Android uses a single event list from the last session only — not true multi-session layout. Major gap.

Summary extraction

Walk items backwards → find last assistant message with content → remove from items (deduplicate) → render below the session, always visible.

⚠️ Platform gap

iOS ships summary extraction. Android does not yet.

User Message Resolution

The user's prompt appears in a bubble above the timeline. It resolves through a priority chain:

1
Session properties

If the session has eventType (e.g., issue_created) or eventContent, use that.

↓ not found?
2
Event stream fallback

Extract the first user.message event. Suppress it inline to avoid duplication.

↓ not found?
3
No bubble

Valid for system-triggered sessions (scheduled, API).

⚠️ Platform gap

Android drops eventType, eventUrl, and resourceNumber during mapping — can only resolve from the event stream, not session properties.
