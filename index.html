Principles

1. Show the outcome, not the process.

Most users don't care about individual tool calls — they want to know: what did I ask, what happened, did it work? The timeline exists for transparency, not as a primary reading experience.

2. One piece of information, one place.

Never show the same content twice (user message in bubble AND inline, summary inside AND outside the card). Extract and deduplicate.

3. Work bursts collapse, sessions don't.

The flat timeline groups consecutive tool calls into collapsible bursts, but individual sessions are not collapsible units. Session boundaries are marked by lightweight inline dividers, not expandable cards.

4. Platform parity over platform flair.

iOS and Android should make the same structural decisions about what to show, hide, group, and extract. Visual styling can differ; information architecture must not.

5. Terminal sessions are frozen, active sessions are live.

Completed/failed/cancelled/timed-out sessions get summary treatment. In-progress sessions show working state. Never show a "summary" for something still running.

Core Concepts

Task
A user-initiated request to a Copilot agent. A task contains one or more sessions. Think of it as the top-level container — "Fix the auth bug" is a task.
Session
A single agent execution within a task. A task may have multiple sessions if the user starts follow-up work, or if work is triggered from different surfaces (CLI, web, mobile). Each session has its own event stream, state, and lifecycle.
Event
An atomic unit within a session's event stream. Events are the raw data — messages, tool executions, permissions, lifecycle signals. The timeline is built by processing and filtering these events.

Timeline Structure

The timeline is a flat vertical stream. There are no per-session cards or expand/collapse wrappers — sessions flow into each other, separated by lightweight inline dividers.

Single-session task

👤 User Message Bubble
├─ 🔧 Tool call
├─ 🔧 Tool call group (burst)   ← collapsible work burst
├─ 💬 Assistant message
├─ 🔒 Permission request
├─ ⟳ Working indicator           ← only if in_progress
└─ 📋 Summary Message            ← only for terminal sessions

Multi-session task

When a task has 2+ sessions, an inline divider separates them:

👤 User Message Bubble (session 1)
├─ 🔧 Tool call group (burst)
├─ 💬 Assistant message
└─ 📋 Summary Message
─── Continued from CLI · GPT-4.1 · 2m ago ───    ← session divider
👤 User Message Bubble (session 2)
├─ 🔧 Tool call
├─ 💬 Assistant message
└─ 📋 Summary Message
─── Continued from web · GPT-4.1 · 5m ago ───    ← session divider
👤 User Message Bubble (session 3)
├─ 🔧 Tool call group (burst)
└─ ⟳ Working indicator

Session divider format

─── Continued from <origin> · <model> · <relative time> ───
  • Origin: human-readable label mapped from producer — "copilot-cli" → "CLI", "copilot-web" → "web", "copilot-vscode" → "VS Code", "copilot-mobile" → "mobile"
  • Model: which model the session used (e.g., GPT-4.1)
  • Relative time: when the session started (e.g., "2m ago")
  • Only appears between sessions (the first session has no divider)
  • Rendered as a centered text divider with horizontal rules — no card, no border, no tap target

Key layout rules

  • User message bubble sits at the top of each session's events, always visible
  • Summary message appears at the end of a session's events, always visible (terminal sessions only)
  • Working indicator appears at the bottom of the timeline when the latest session is in_progress
  • Session divider appears between sessions — it's a timeline entry, not a structural header
  • Status is communicated through the working indicator (in-progress) and summary (terminal) — NOT the divider
  • Tool call bursts are the only collapsible unit in the timeline

User Message Resolution

The user's initial prompt should always appear in the bubble above the timeline. It resolves through a priority chain:

Priority 1: Session properties

The session object itself may carry context from the trigger:

eventTypeBubble shows
issue_created / issue_assignment"Task started from issue #N"
pull_request_comment"New comment on pull request #N"
pull_request_review"New review on pull request #N"
workflow_run_failed"Workflow run failed"
interval_triggered"Scheduled task"
slack"Slack message"
api_call_received"API call received"
(none, but eventContent exists)Raw eventContent text

Priority 2: Event stream fallback

If session properties don't resolve a bubble, extract the first user.message event from the event stream.

  • Must not be empty after trimming
  • Must not contain XML system tags (<pr_title>, etc.)
  • When used, suppress the inline user message to avoid duplication

Priority 3: No bubble

If neither resolves, the session has no user message bubble. This is valid for system-triggered sessions.

Deduplication rule: When the bubble is populated from eventContent or the event stream fallback, the corresponding inline user.message in the timeline is suppressed. When the bubble is synthesized from eventType/eventUrl metadata (e.g., "Task started from issue #42"), the inline user message is kept — because the bubble shows context, not the prompt itself.

Tool Call Grouping

The problem

A typical agent session has dozens of tool calls. Showing each one flat creates an unreadable wall of items.

The solution: Work bursts

Consecutive tool calls are grouped into collapsible "bursts."

Rules

  1. Minimum group size: 3. Fewer than 3 consecutive tools render flat (no grouping).
  2. "task" tool breaks the burst. Background agent launches are structurally significant — show them standalone.
  3. User messages break the burst. A user message means a new conversational turn.
  4. Permission requests and prompts break the burst. These are interaction points that shouldn't be hidden.
  5. Intermediate assistant messages become "thinking" items inside the group. The final assistant message after a burst stays outside as a visible response.
  6. Burst header: "Used N tools" with aggregate status (running/succeeded/failed).
  7. Duration: Computed from first to last tool in the group.
  8. Auto-expand: The last burst in the timeline starts expanded if:
    • Session is in-progress (show what the agent is actively doing)
    • No trailing message follows (don't hide the outcome)
⚠️ Platform Gap

iOS implements this via groupToolCallBursts() in the view model builder. Android groups consecutive same-name tools into ToolCallGroup but does not yet implement the broader work burst pattern. This is a parity gap to close.

Summary Extraction

What

The last assistant message with non-empty content is extracted from the timeline and displayed at the end of the session's event block, always visible.

When

Only for terminal sessions (completed, failed, cancelled, timed_out). Never for active sessions.

How

  1. Walk the items array backwards
  2. Find the last .assistant message with non-empty content
  3. Remove it from items (prevents duplication)
  4. Store as summaryMessage on the session view model
  5. Render as plain markdown below the card

Why

Users need to see "what happened" at a glance. The summary is the answer to "did it work?" — it should be visible in the flat timeline without any interaction.

⚠️ Platform Gap

iOS implements this (PR #24323). Android does not yet. Parity gap to close.

Multi-Session Behavior

When do multiple sessions happen?

  • User sends a follow-up message after a session completes
  • User triggers work from a different surface (CLI session + web session)
  • Session handoff between clients

Layout

The timeline is flat and continuous. Sessions are separated by inline dividers, not cards:

Task Header
  👤 User bubble (session 1, oldest)
  ├─ Timeline events
  └─ Summary
  ─── Continued from web · GPT-4.1 · 12m ago ───
  👤 User bubble (session 2)
  ├─ Timeline events
  └─ Summary
  ─── Continued from CLI · GPT-4.1 · 3m ago ───
  👤 User bubble (session 3, latest)
  ├─ Timeline events
  └─ Working indicator (if active)

Rules

  1. Flat timeline, no per-session collapse. Sessions are not expandable/collapsible units. The entire timeline reads top-to-bottom as a continuous stream.
  2. Session dividers appear between sessions. The first session has no divider. Each subsequent session is preceded by an inline divider showing origin, model, and relative time.
  3. Each session is self-contained. Its bubble, events, summary, and state come from its own event stream. No cross-session leaking.
  4. Chronological order. Sessions render in creation order, oldest first.
  5. Work bursts are the collapsible unit. Within a session, consecutive tool calls still group into collapsible bursts — but the session boundary itself is just a divider, not a card.

Session Origins — Why Things Look Different

Sessions arrive from different surfaces, and each populates different properties. This is why "the same task" can look different depending on where it was started.

OrigineventContenteventTypeeventUrlUser bubble source
Mobile (Agents Panel)User's typed prompt——Session property
Web (github.com)Issue/PR contextissue_created, pull_request_comment, etc.GitHub resource URLSession property
CLI (Copilot CLI)Often empty——Event stream fallback
Desktop (VS Code)Often empty——Event stream fallback
Scheduled / API—interval_triggered, api_call_received—Constructed label

This is why PR #24321 exists — CLI and desktop sessions had no user bubble because they don't populate eventContent. The fallback to event stream closes this gap.

⚠️ Platform Gap

Android drops eventType, eventUrl, and resourceNumber when mapping session models (AgentTasksResponseMapper), so it cannot currently resolve bubbles from session properties — only from the event stream.
